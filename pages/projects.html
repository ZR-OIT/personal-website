Embedded Junior Project -
Electronic Directory Board

The embedded junior project at Oregon Tech is a team-based project that includes designing and buidling an embedded system to solve a particular problem. Myself and three others decided to build an electronic directory board to keep track of classroom and lab schedules.

Why a directory board? Many students need to use classrooms or labs outside of their scheduled course times. However, these rooms could be occupied by a professor and/or their students at any given time, so awkward interruptions can happen when walking into a room to use the equipment.

With an electronic directory board posted in the main lobby, students and professors could easily see which rooms are open for use by looking at the status LEDs. With a combination of a pre-programmed schedules and motion sensors, the directory can show if rooms are open or booked as well as if anyone is currently in the room.

Once we had the idea, we researched and compared many different components including microcontrollers, shift registers, power supplies, and LEDs. We eventually decided to use the ESP32 Arduino because it was low-cost and had built in Wi-Fi. With all the parts chosen, I made basic schematics in Microsoft Visio to show how the circuits are assembled.

As soon as we had the parts in our hands, we tested everything individually before assembling the circuits. It was important to make sure that all LEDs, shift registers, and motion sensors functioned as expected, otherwise we would have a hard time debugging later on. Aside from an issue with some loose wires, the circuitry went together and worked well without much trouble. An Arduino UNO was used for testing since we were already familiar with it.

While working on the circuitry, we also had to get the web server up and running on the ESP32. Initially, another team member was tasked with building the website and server. However, the team member was struggling to make substantial progress, so I took over the web server development and moved them over to designing a housing. After a long weekend, I got a simple website implemented that could control the hardware.

A library called ESPAsyncWebServer was used to implement the web server logic. The HTML and CSS files were flashed to the non-volatile memory of the ESP32 using SPIFFS. The HTML files are altered before they are served to the user based on the current day and time. We also had to consider multiple people logging into the website at once, so we captured the IP of the first authorized user and prevented other logins until the first authorized user disconnects. There was dicussion of implementing this system in the building, but COVID-19 forced the campus to close.

Embedded Senior Project -
Bluetooth OBD-II diagnostic tool with touchscreen

The embedded senior project at Oregon Tech is a solo project. I struggled to come up with an idea at first, but eventually decided on building a Bluetooth OBD-II diagnostic tool with a touchscreen. An OBD-II diagnostic tool is used to interface with vehicles to diagnose check engine lights or to obtain data such as temperatures, pressures, speed, and RPM.

I wanted to build this tool for a couple different reasons; primarily because I have always liked working on cars. I also wanted to learn more about schematic capture, the CAN bus, Bluetooth, and Raspberry Pi/Linux development. On top of that, I wanted to design and 3D print a housing, so I created prototype renders in SketchUp. The entirety of this project was developed in a remote environment during the peak of the pandemic, so access to lab equipment and other resources was limited.

Before I could start making schematics and building the circuitry, I had to research the OBD-II specifications and the different components I would need to interface with the port. I determined I would use the ELM327 OBD interpreter IC with an ESP32 to interface with the OBD-II port, and a Raspberry Pi 4 for the handheld unit. I chose the ELM327 because it is widely supported and well documented.

With help from the ELM327 datasheet, I made schematics for the OBD-II port module and the handheld module using KiCAD. I also created symbols for ICs that were not included with KiCAD. For development of the OBD-II transceiver, I planned to use Visual Studio Code with the official ESP-IDF extension from espressif to write in C/C++. The handheld device would use Python scripts for the GUI and Bluetooth data transfer.

With the schematics made, I could begin incrementally assembling and testing the circuitry. First, I tested all components individually to make sure they were within tolerance and/or functioned correctly, including LEDs, resistors, and a button. Then, I added all the components needed for power delivery. I wanted to make sure that the 12VDC from the car battery was being reduced to 5V and 3.3V properly before adding other components.

Once the rest of the OBD-II transceiver circuit was assembled, I used an oscilloscope to make sure the 3.3V and 5V logic level translation was working correctly (the ESP32 uses 3.3V logic while the ELM327 uses the older 5V logic.) Once I was sure everything worked as expected, I tested the circuit on the car with a wired serial connection before attempting Bluetooth communication.

With the circuitry assembled, I started working more on the handheld Raspberry Pi module. Previously, I researched many different ways to power the Pi, and eventually decided on a uninterruptable power supply (UPS) hat. Using this method, the Pi could safely shutdown automatically when the battery voltage dropped too low, and the rechargeable battery would last at least 2 hours.

Initially, I decided to use default Raspberry Pi OS, but later switched to the Lite version after realizing I did not need everything that was included with the default version. I also had to make some minor tweaks to the OS to allow the button to turn the Pi on and off, as well as to allow a GUI on the Lite version of Raspberry Pi OS.

At this point, the end of the academic year was approaching, and I still didn't have Bluetooth data transfer or a GUI. Using the examples included in the Arduino IDE, I tested to see if the ESP32 would even connect to the Raspberry Pi with Bluetooth. The connection worked, but it was just a connection with no data transfer. I also got started on the general layout of the user interface using PySimpleGUI.

I presented my project as it was, demonstrating the communication with the car through a wired serial connection. There was enough functionality and progress made to consider the project a success, but it still didn't live up to my expectations.

Looking back, the project was a overly ambitious given the circumstances including working remotely with limited resources, struggling with mental health, and having to stay on top of other courses. I wanted to design and build circuits, design and 3D print housings, implement Bluetooth logic on two different platforms, design and build a touchscreen user interface, and more - all on my own within 9 months. However, I still learned a lot of useful things over the course of the project including researching and comparing components, schematic capture, how the OBD-II port and CAN bus work, how Bluetooth works, and developing on Linux/Raspberry Pi.
